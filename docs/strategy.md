# 当前策略说明书

## 1. 策略目标

当前策略采用“送达优先，安全拿分”的原则：

1. 第一目标：完成 S14 朱雀门验核并在 S15 兴庆宫交付。
2. 第二目标：在不影响交付的时间窗内争取皇榜任务分，优先接近 90 分任务里程碑。
3. 第三目标：通过资源、窗口出牌、障碍处理和终局急策提高交付速度与稳定性。

策略不追求全量博弈最优，而是先保证合法、可运行、可复盘、可扩展。

## 2. 当前策略管线

`strategies/factory.py` 中的默认顺序为：

```text
CombatStrategy
  -> SquadStrategy
  -> TaskStrategy
  -> ResourceStrategy
  -> RushStrategy
  -> GuardStrategy
  -> DeliveryStrategy
```

含义：

- 如果正在窗口争夺，优先出窗口牌。
- 如果普通阶段还有小分队人手，提前探路关键处理点；主线未来有障碍时优先远程清障。
- 如果当前节点有安全可做的任务，优先处理任务。
- 如果可以合法使用或领取关键资源、情报，则处理资源。
- 如果冲刺阶段在路线边且时间紧，考虑疾行令。
- 如果已经达到 90 分任务基础线且还有足够交付余量，在对手未来会经过的高价值节点保守设卡。
- 其他情况进入主线交付策略。

管线不是简单“第一个动作独占整帧”，而是按动作类别合并：主车队、小分队、窗口出牌、急策额度各自最多 1 个。同类动作会按优先级裁决，高优先级动作可以替换低优先级动作，例如 `CLEAR` 会压过 `USE_RESOURCE INTEL`，关键资源领取也可以压过普通 `MOVE`；不同类动作会同帧提交，例如 `MOVE` 可以和 `SQUAD_SCOUT` 一起发送。

## 3. 路线策略

默认 `auto` 路线分两层：

1. 先保证从 S01 到 S02，并完成 S02 前段交接。
2. S02 之后不再强制水路，而是按当前地图、天气、固定处理耗时、任务收益、道路障碍、清障残留和敌方设卡做动态寻路。

当前第一轮地图在普通天气下，动态代价通常会自然选择水路：

```text
S01 -> S02 -> S04 -> S05 -> S09 -> S10 -> S11 -> S12 -> S13 -> S14 -> S15
```

显式 `--route-profile=first-round-water` 会强制使用上述水路主线；`first-round-safe` 是旧参数名，当前等价于水路主线；`generic` 和默认 `auto` 都交给动态图搜索。第一轮/决赛地图变化时，策略不会把路线写死到交付逻辑里。

主线覆盖的固定处理点按实际经过路线触发：

- S02：前段交接，`PROCESS`
- S04：登船处理，`PROCESS`
- S05：水路换运，`PROCESS`
- S11：入关交接，`PROCESS`
- S13：宫前交接，`PROCESS`
- S14：宫门验核，`VERIFY_GATE`
- S15：终点交付，`DELIVER`

在 S09 面对 S10 阻挡时：

- 如果 S10 有道路障碍，当前位置 S09 与 S10 相邻，直接提交 `CLEAR S10`；清障完成后主车队仍在 S09，下一帧重新读取状态后再 `MOVE S10`。
- 如果 S10 是敌方有效设卡，提交 `BREAK_GUARD S10`；攻破后仍在 S09，再提交 `MOVE S10`。
- 如果果品或时间不足以安全处理阻挡，则提交 `FORCED_PASS S10` 兜底通过。

## 4. 任务策略

任务策略由 `planning/tasks.py` 和 `strategies/tasks.py` 共同实现。

选择原则：

- 只处理 `active=true`、未完成、未失败、未过期、未被其他玩家保护的任务。
- T04 可以在障碍节点或相邻节点处理，其余任务必须到任务节点。
- T06 需要持有快马或短程马，否则不接。
- 任务完成时间必须早于过期时间。
- 完成任务后仍要满足“剩余交付估算 + 安全余量 < 600”。
- 当前任务截止帧为 370，安全余量为 75 帧。
- 任务分达到 110 后停止继续追任务。
- 任务排序会优先考虑 90/110/130 分里程碑，其次看任务分值、模板优先级、单位耗时收益和绕路成本。

拒绝保护：

- 如果某个 `CLAIM_TASK` 被服务端拒绝，该任务实例会进入本局拒绝集合。
- 后续任务策略和交付策略都不会再选择该任务，避免重复拒绝导致主线停滞。

## 5. 资源策略

资源策略由 `strategies/resources.py` 实现。

领取资源：

- 只在 `IDLE`、非宫门、非终点、交付仍安全、且当前帧小于 330 时领取。
- 当前关注资源：`ICE_BOX`、`FAST_HORSE`、`SHORT_HORSE`、`PASS_TOKEN`、`OFFICIAL_PERMIT`、`INTEL`、`BOAT_RIGHT`。
- 每类资源有库存上限和安全余量：冰鉴、马类、文书和情报优先级更高；`BOAT_RIGHT` 只在 S04 这类码头语境下保守领取。
- 每个节点的每种资源只尝试领取一次，避免库存或窗口异常时重复卡住。

使用资源：

- 路线边 `MOVING/WAITING`：只允许使用 `FAST_HORSE`、`SHORT_HORSE`。
- 停靠节点 `IDLE`：允许使用 `ICE_BOX`，但必须鲜度大于 0、持有冰鉴、交付仍有余量；高温或冲刺阶段会更早保鲜。
- 停靠节点 `IDLE`：允许使用 `INTEL`，目标必须在 15 格距离内，优先给当前固定处理点，再给 S11/S13/S14/S04/S05 等关键节点加探路标记。
- 固定处理点不提前使用马，避免马类 buff 被处理读条浪费。
- 宫门 S14 和终点 S15 停靠时不使用资源，避免抢占验核或交付动作。
- 超过任务截止后，不再为了 T06 保留马类资源，优先服务送达。

本轮修复重点：

- `WAITING` 已纳入路线边状态。
- `WAITING/MOVING + ICE_BOX` 会返回空动作，不会再触发 `MOVING_ACTION_FORBIDDEN` 循环。

## 6. 交付策略

交付策略由 `strategies/delivery.py` 实现，是最终兜底策略。

主线流程：

1. 如果在 S15 且已验核、好果大于 0、鲜度大于 0：立即 `DELIVER`。
2. 如果在 S15 但未验核：返回 S14。
3. 如果在 S14 且未验核：
   - 非 `RUSH` 阶段：空动作等待宫宴冲刺。
   - `RUSH` 阶段：提交 `VERIFY_GATE`。
   - 只有 `breakOrderReady=true`、急策未用、成本足够且收益明确时，才绑定 `BREAK_ORDER` 缩短验核。
4. 如果在普通固定处理点：提交 `PROCESS`。
5. 从 S01 出发时，先去 S02 完成前段交接。
6. 如果还有安全任务目标：移动到任务目标或在当前节点 `CLAIM_TASK`。
7. 否则向 S14/S15 主线目标移动。

固定处理保护：

- S02、S04、S05、S11、S13 每次重新到达都需要重新处理。
- 处理完成事件只标记当前停靠访问有效；离开后再回来会清除旧完成标记。
- S04 登船处理统一提交 `PROCESS`，与任务书中固定处理动作一致。

## 7. 障碍和设卡处理

目标节点有道路障碍时：

1. 如果存在可做的 T04，优先 `CLAIM_TASK`，同时拿任务分并清障。
2. 如果没有 T04、好果充足、交付余量足够且时间不晚，提交 `CLEAR`。
3. 如果时间紧、好果不足或已经接近终局，提交 `FORCED_PASS`，优先保证通过。

目标节点有敌方设卡时：

1. 如果投入最多 2 好果 / 2 坏果可以一次攻破，且目标没有道路障碍，使用 `BREAK_GUARD`。
2. 如果设卡防守过高、目标同时有道路障碍，或交付优先级更高，使用 `FORCED_PASS`，避免低攻坚值反复失败后休整。
3. 如果前一帧 `MOVE` 被隐藏设卡拒绝，下一帧会记住该目标并改用 `FORCED_PASS`。

这套逻辑的核心是避免停在阻挡前反复提交会失败的 `MOVE`。

## 8. 窗口争夺策略

窗口策略由 `strategies/combat.py` 实现。

如果 `contests[]` 中存在自己参与、未解决、非 `SUPPRESSED` 的窗口，提交一张窗口牌：

1. 关键窗口且有护卫行动点：`BING_ZHENG`
2. 鲜度至少 80 且好果较多：`XIAN_GONG`
3. 有过所或官凭：`YAN_DIE`
4. 资源/任务窗口仍有护卫行动点：`BING_ZHENG`
5. 有马类资源或移动 buff：`QIANG_XING`
6. 否则：`ABSTAIN`

窗口出牌属于独立窗口动作，允许在移动、等待、窗口争夺等状态下提交。

## 9. 主动设卡策略

主动设卡由 `strategies/guard.py` 实现，属于第一版保守干扰能力。

当前只在以下条件同时满足时提交 `SET_GUARD`：

- 普通阶段，主车队 `IDLE`，当前位置是当前地图上的高价值设卡点，例如 S10、S11、S14，或节点类型为关键关隘/宫门。
- 当前任务基础分已经达到 90，优先保证送达分和用时分吃满。
- 好果足以支付设卡基础成本和保底余量，且估算交付仍保留安全余量。
- 当前节点没有己方有效设卡，也没有敌方设卡。
- 至少一个对手的预计后续路线会经过当前节点，且预计到达时间足够让设卡生效。
- 当前最多保留 2 个己方有效设卡；如果我们公开总分明显领先，则避免主动制造对手可提款的悬赏风险。

默认设卡只投入 `extraGoodFruit=0`；如果已经达到 110 任务基础分、好果充足且位于 S10 关键关隘，才会额外投入 1 篓好果提高防守值。目标是制造轻量干扰和潜在悬赏机会，不为了主动卡人牺牲交付安全。

## 10. 小分队策略

小分队策略由 `strategies/squad.py` 实现，动作独立于主车队动作。

当前规则：

- 只在普通阶段派出新小分队，冲刺阶段停止新增小分队动作。
- 每帧最多提交 1 个小分队动作，由管线类别合并保证。
- 如果主线未来路径上有可见敌方设卡且人手至少 2，优先 `SQUAD_WEAKEN`。
- 如果主线未来路径上有可见道路障碍且人手至少 2，其次 `SQUAD_CLEAR`。
- 否则按 `S04 -> S05 -> S11 -> S13 -> S14` 依次 `SQUAD_SCOUT`，提前给登船、水路换运、入关、宫前和宫门验核加探路标记。
- 已派出或已存在己方探路标记的目标不会重复派出。

小分队策略的目标是提前降低固定处理/宫门验核耗时，同时不占主车队动作，避免为了加分拖慢送达。

## 11. 终局急策

当前实现三类急策使用：

- `RUSH_SPEED`
  - 由 `strategies/rush.py` 控制。
  - 只在 `phase=RUSH`、路线边 `MOVING/WAITING`、未使用过急策、无移动 buff、手里没有马类资源、好果大于 2、估算交付偏紧时提交。
  - 作用是提高路线边推进速度，帮助压缩最后送达时间。

- `RUSH_PROTECT`
  - 由 `strategies/rush.py` 控制。
  - 只在冲刺阶段、停靠节点、未使用过急策、没有护果 buff、后续交付仍有明显路程、鲜度已经危险时提交。
  - 高温天气下会提前触发，目标是避免终点前鲜度坠穿导致无法交付或品质分大幅损失。

- `BREAK_ORDER`
  - 由 `DeliveryStrategy` 绑定到 `VERIFY_GATE`。
  - 只在冲刺阶段、`breakOrderReady=true`、急策未用、成本足够、且时间极晚或坏果足够时使用。
  - 不会单独发送 `BREAK_ORDER`，避免非法动作。

## 12. 状态限制覆盖

| 任务书限制 | 当前处理 |
|---|---|
| `MOVING` 不能执行站点动作 | `MAIN_ACTION_BUSY_STATES` 阻止任务和交付策略；资源策略只允许马类。 |
| `WAITING` 是路线边主动等待 | 与 `MOVING` 同样作为路线边状态处理，不再执行冰鉴、任务、处理。 |
| 处理、验核、休整、强制通行中动作不打断当前流程 | 策略返回空动作，发送 `actions: []`。 |
| 路线边只能用马类资源和疾行令 | 资源策略只给马；急策策略只在路线边给 `RUSH_SPEED`。 |
| 情报只能停靠节点使用且距离不超过 15 | 资源策略只在 `IDLE` 节点使用 `INTEL`，并用地图坐标校验距离。 |
| 冰鉴鲜度为 0 时会被拒 | 策略要求 `freshness` 非 0。 |
| S15 安全区动作受限 | 资源和任务策略不在 S15 停靠时动作；交付策略只交付或返回 S14。 |
| 未验核不能交付 | S15 未验核时返回 S14；S14 冲刺阶段验核。 |
| 破关令不能单独发送且需要准备态 | 只作为 `rushTactic` 绑定到 `VERIFY_GATE`，且要求 `breakOrderReady=true`。 |
| 固定处理完成前不能离站 | 交付策略先处理固定流程，再移动。 |
| 离开处理点后再次到达要重新处理 | 策略按停靠访问重新清除固定处理完成标记。 |
| 任务被保护/过期/拒绝 | 规划层过滤；拒绝任务加入黑名单。 |
| 道路障碍阻挡 MOVE | T04、CLEAR、FORCED_PASS 三层处理。 |
| 敌方设卡阻挡 MOVE | 能一次攻破才 BREAK_GUARD，否则 FORCED_PASS；小分队可提前 SQUAD_WEAKEN。 |
| 清障残留通行税 | 路由代价会计入对方清障残留税，必要时绕行。 |
| 每帧小分队动作最多 1 个 | 策略管线按 `squad` 类别合并并跳过同类重复动作。 |
| 已交付后不能乱动 | 各策略看到 `delivered` 后返回空动作。 |

## 13. 送达保证边界

在以下前提下，当前策略会优先保证最终抵达并交付：

- 地图从起点到 S14/S15 连通。
- 服务端字段与任务书/协议一致。
- 本队仍有好果大于 0 且鲜度未归 0。
- 对手干扰没有造成长期无法通过的极端阻塞。

策略层的保证手段是：

- 370 帧后不再主动追远任务。
- 任务、资源动作都受交付估算和安全余量约束。
- 默认先完成 S02 前段交接；之后由动态路由决定走水路、陆路、任务绕行或避开阻挡。
- 路线边返回空动作推进，不提交主动 `WAIT`。
- 阻挡前不重复普通 `MOVE`，改用清障、攻坚或强制通行。
- 小分队和窗口牌作为独立类别合并，不抢主车队动作。
- 宫门验核即使不绑定破关令也会执行，破关令前提不满足时不会阻塞 `VERIFY_GATE`。
- S14/S15 有最高优先级交付逻辑。

无法由客户端单方面保证的情况：

- 地图本身不连通。
- 好果或鲜度已经不足以满足交付条件。
- 服务端长期返回业务拒绝但公开状态不体现原因。
- 对手窗口博弈持续压制，导致强制通行或处理长期失败。

## 14. 后续优化方向

建议按收益优先级继续扩展：

1. 完整比赛仿真器：用真实地图、天气、任务刷新和障碍生成模拟 600 帧。
2. 更精细的小分队策略：根据预计到达时间、在途小分队和天气延迟选择探路/清障目标。
3. 更精细情报策略：结合当前库存、探路标记剩余、S14 等待时间决定是否给宫门预铺。
4. 更精细任务组合：按“任务分里程碑收益 + 路线绕行成本 + 过期风险”做组合优化。
5. 对手建模：根据对手位置预测窗口争夺、设卡和障碍抢占。
6. 动作拒绝恢复表：不同 `errorCode` 映射到冷却、换路、强制通行或放弃目标。
7. 完整天气规划：把未来天气预报也纳入水路、山路或官道切换。
